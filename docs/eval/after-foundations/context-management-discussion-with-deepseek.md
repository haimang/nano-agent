### 📋 完整功能与价值组合总表

以下按你提出的核心架构分层，将每个功能点及其对应的价值一一列出。

---

#### 🏛️ 一、核心架构理念

| 功能 / 特性 | 价值主张 |
| :--- | :--- |
| **上下文即负债** | 主动限制使用量在 300k–400k 高效区间，避免超额上下文稀释模型注意力、降低输出质量、浪费 API 额度。 |
| **从被动压缩到主动管理** | 用户成为上下文的架构师，拥有定义、保护、淘汰上下文的决策权，而非被动接受黑盒压缩。 |
| **结构化分层管理** | 将上下文按性质隔离，不同层级采用不同生命周期策略，核心信息永不丢失，冗余信息精准剔除。 |

---

#### 🧱 二、上下文分层模型

| 层级 | 功能描述 | 价值主张 |
| :--- | :--- | :--- |
| **L0: System Prompt** | 锁定层，永久固定，不参与任何压缩。 | 确保 AI 角色定义、核心行为准则在整个会话中绝对稳定，不会因压缩而“性格突变”。 |
| **L1: Memory Prompt** | 锁定层，存储项目知识、用户偏好、编码规范等持久记忆，不参与压缩。 | 跨会话保持项目上下文一致性，避免每次对话都要重新“教育” AI，显著提升长周期任务效率。 |
| **L2: Working Memory** | 动态可压缩层，存储对话历史与工具调用结果，按策略淘汰或压缩。 | 保留近期交互细节的同时，控制上下文膨胀，为模型留出充足注意力空间处理当前问题。 |
| **L3: External Knowledge** | 按需检索层，文件内容、文档等通过 RAG 注入，不常驻上下文。 | 用精准检索替代全量携带，既保证知识可及性，又避免无关内容占用宝贵 Token。 |

---

#### ⚙️ 三、压缩机制

| 功能 / 特性 | 价值主张 |
| :--- | :--- |
| **异步压缩** | 压缩任务在后台 Service Worker 执行，主推理流程完全不阻塞。用户不会感知到“压缩中，请稍候”的卡顿，心流体验连贯。 |
| **可自定义压缩目标层** | 用户可指定仅压缩 L2 层，L0 和 L1 层完全跳过。核心规则和项目记忆得到绝对保护，压缩安全性可控。 |
| **本地 MoE 模型压缩** | 通过 Hook 挂载 `llama.cpp` 运行的本地 MoE 模型（如 Qwen3-Coder-30B-A3B）执行压缩。压缩过程零 API 费用，且压缩逻辑可完全自定义。 |
| **PreCompaction Hook** | 压缩前触发，可获取待压缩上下文，并返回自定义摘要。用户可在压缩前插入任意预处理逻辑（如调用本地 LLM、数据清洗），实现“我的压缩我做主”。 |
| **AfterCompaction Hook** | 压缩后触发，用于审计、日志记录或触发后续动作。提供压缩过程的可观测性，便于调试和优化压缩策略。 |
| **临时禁止压缩** | 重要长推理前可通过开关或 Hook 返回值暂时禁用压缩。确保关键任务期间上下文绝对完整，避免因压缩丢失中间推导状态。 |
| **文件去重与缓存** | 已读取过的文件内容以路径 + Hash 为 Key 缓存，重复引用时只插入引用标记。大幅减少因重复读取同一文件造成的 Token 浪费，尤其利好调试迭代场景。 |

---

#### 📊 四、预算控制与限制

| 功能 / 特性 | 价值主张 |
| :--- | :--- |
| **自定义 Buffer Size** | 用户可配置硬上限（如 300k）和软上限（如 80% 触发压缩）。将上下文用量牢牢锁定在模型表现最佳的“甜点区”，同时预留缓冲空间应对突发需求。 |
| **动态调整上限** | 支持通过环境变量或命令临时放宽限制（例如需要 1M 空间时）。赋予用户“按需扩容”的灵活性，常规任务保持克制，复杂任务一键释放潜力。 |
| **Token 用量实时监控** | 提供当前上下文各层 Token 占用可视化。用户对上下文消耗有清晰感知，变“黑盒消耗”为“白盒预算”，便于决策何时手动干预。 |

---

#### 🔌 五、Hook 扩展体系

| Hook 名称 | 功能描述 | 价值主张 |
| :--- | :--- | :--- |
| **SessionStart** | 会话启动时触发，用于初始化分层、注入 RAG 检索结果、设定预算。 | 自动化上下文初始化，让每个新会话都从“最佳配置”出发，减少手动重复设置。 |
| **PreCompact** | 压缩前触发，可调用本地 LLM 生成摘要或跳过压缩。 | 将压缩决策权完全开放给用户，是实现智能压缩、成本优化的核心入口。 |
| **AfterCompact** | 压缩后触发，用于验证结果、记录审计日志。 | 为压缩质量提供可追溯的审计线索，便于持续优化压缩策略。 |
| **MemoryInject** | 用户主动调用，向 L1 Memory 层注入指定文本或文件，并标记为受保护。 | 用户可随时向 AI 的“长期记忆”写入关键信息，确保其在后续所有交互中都被牢记，无需重复提醒。 |
| **PreToolUse / PostToolUse** | 工具调用前后触发，可用于文件读取去重、缓存管理。 | 在工具层面拦截冗余操作，从源头减少不必要的上下文开销，进一步提升效率。 |

#### 🔗 六、外部接口与集成

| 功能 / 特性 | 价值主张 |
| :--- | :--- |
| **RAG 接口** | 支持向量检索，按需从知识库拉取内容注入 L3 层或 Memory。 | 将企业私有知识库与 AI 工作流无缝衔接，让 AI 在编码时可实时引用最新文档、历史代码库，提升生成准确性。 |
| **KV 存储 (JSON 结构化管理)** | 上下文分层以 JSON 形式存储在 Redis / LevelDB 中，不同层级 Key 隔离。 | 提供原子化、可编程的上下文操作能力，TypeScript 预处理层可直接读写、清洗、转换上下文数据，为高级用户提供最大操控自由度。 |
| **Service Worker 解耦架构** | Hooks 逻辑运行在独立进程，与主 Agent 通过 IPC/HTTP 通信。 | 主推理流程零侵入、零阻塞；Worker 崩溃不影响主会话；扩展功能可独立开发、部署、调试。 |
| **本地 LLM 服务接口** | 通过 `llama.cpp` Server 提供 OpenAI 兼容 API。 | 压缩、摘要、预处理等任务可卸载到本地模型，在保证质量的同时将 API 成本压至最低。 |
| **S3 / 文件系统持久化** | Memory 层数据先在 KV 缓存热读，后台异步持久化到 S3 或本地文件。 | 兼顾高速读写与长期可靠存储，Memory 数据可跨会话、跨设备恢复，构建真正持久的 AI 记忆。 |
